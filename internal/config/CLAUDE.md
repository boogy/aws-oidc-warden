# Config — Configuration Management

Extends [../../CLAUDE.md](../../CLAUDE.md). Viper-based loading, validation, and issuer-bound subject/condition matching (`config.go`).

## Loading

- `NewConfig()` loads once (`sync.Once`) and returns the shared instance.
- Precedence: `AOW_`-prefixed env vars > config file (YAML/JSON/TOML) > defaults.
- Nested keys map to underscores: `AOW_CACHE_TTL` → `cache.ttl`. See `example-config.yaml` for the full reference.

## Key structures & methods

- `Config` — `issuers` (`[]IssuerConfig`, v2 multi-issuer model; no legacy single `issuer`/`audience`/`audiences` fields), `default_issuer`, `role_mappings`/`role_groups`/`role_sets`, `config_fragments`/`config_fragment_checksums`, S3 config/policy buckets, logging, `cache`, `TagAuth`, `JWTValidation`, plus top-level hardening knobs.
- `IssuerConfig` — one trusted issuer: `issuer` (exact `iss` match, no normalization), `provider` (`"github"` = native claim struct unmarshal, `"generic"` = mapped-only via `claim_mappings`; default `"generic"`), `audiences` (ANY-match, ≥1 required), `jwks_uri` (optional override, skips OIDC discovery), `claim_mappings` (canonical field ← raw claim; may never target a JWT-reserved claim: `iss/aud/exp/nbf/iat/sub`), `required_claims`, `session_tags` (STS tag key ← raw claim; keys must match `[A-Za-z0-9 _.:/=+@-]{1,128}`). Non-`github` issuers must set `claim_mappings.subject`.
- Zero-config seed: if no config file/source is found (`viper.ReadInConfig()` returns `ConfigFileNotFoundError`) and `issuers` is still empty after unmarshal, `LoadConfig()` seeds a single default GitHub Actions issuer (`defaultGitHubIssuer()`). If a config source *is* present but defines no issuers, `Validate()` hard-fails — the seed never masks a misconfigured file.
- Hardening knobs (top-level, all optional): `jwt_leeway` (default 30s, hard max 120s — `Validate()` rejects anything higher), `max_token_lifetime` / `max_token_age` (default 1h each; 0/unset applies the default, there is no "no cap" value), `max_token_bytes` (default 8192), `jwks_refetch_cooldown` (default 60s), `allow_insecure_issuers` (default false, dev-only escape hatch for `http://` issuer/jwks_uri). Env: `AOW_JWT_LEEWAY`, `AOW_MAX_TOKEN_LIFETIME`, `AOW_MAX_TOKEN_AGE`, `AOW_MAX_TOKEN_BYTES`, `AOW_JWKS_REFETCH_COOLDOWN`, `AOW_ALLOW_INSECURE_ISSUERS`.
- Logging/audit knobs: `log_level` (`debug`/`info`/`warn`/`error`, default `"info"`; `Validate()` rejects anything else), `log_claim_values` (default `false` — suppresses claim VALUES in structured logs/audit records, keeping names/decision/reason), `audit_required` (default `false` — when `true`, requires `log_to_s3=true` + `log_bucket` set, and makes the audit-record write durable/fail-closed before an allow decision returns credentials). Env: `AOW_LOG_LEVEL`, `AOW_LOG_CLAIM_VALUES`, `AOW_AUDIT_REQUIRED`.
- `Validate()` on `issuers`: at least one required; exact-match duplicate issuer URLs rejected; each needs ≥1 audience; `provider` restricted to `github`/`generic`; `claim_mappings` reserved-claim and `session_tags` charset checks run per issuer.
- `JWTValidation` — `Mode` (`"self"` default, `"apigw"`, `"alb"`) and `ALBExpectedSigner` (ARN of the trusted ALB; **required** in `alb` mode — `Validate()` fails without it, preventing cross-ALB token injection). Env: `AOW_JWT_VALIDATION_MODE`, `AOW_JWT_VALIDATION_ALB_EXPECTED_SIGNER`. `Validate()` rejects unknown modes.
- `RoleMapping` — `subject` pattern, `roles` (ARNs or `@role_set` names), optional `issuer` (defaults to `default_issuer`), optional `session_policy`/`session_policy_file`, optional `conditions`.
- `Condition` — named fields `branch`, `ref`, `ref_type`, `event_name`, `workflow_ref`, `environment` (checked against the raw `runner_environment` claim), `actor_matches`, plus an `Extra map[string]string` for any other raw claim by name. All compile through the same anchored-regex mechanism (`compileCondition`/`compileAnchoredCondition`) into one AND'd list (`satisfiesConditions`) — a named field and an `Extra` entry targeting the same underlying claim both apply and must both match.
- `RoleGroup`/`RoleGroupDefaults` — one entry expands to one `RoleMapping` per `subjects[]` entry, sharing `issuer` + `defaults` (roles/conditions/session policy).
- `RoleSets map[string][]string` — named, reusable role-ARN lists; referenced from a mapping's `roles` as `"@name"` (`resolveRoleSet` strips the `@` before the map lookup — map keys themselves must NOT include `@`).
- `Validate()` — compiles all regex patterns once; subject/condition patterns auto-anchored `^(?:pattern)$`; resolves `role_sets`/`role_groups` into `effective` `RoleMapping`s and builds the per-issuer owner-bucketed `authzIndex` (`index.go`) over them. Bucketing (`exact`/`byOwner`/`any`) is performance-only, but that rests on **two** properties, not one: re-verifying every candidate against its own compiled pattern rules out false *positives*, while soundness of the bucket assignment itself rules out false *negatives* — a mapping filed in a bucket `candidatesFor` never consults is simply never seen, and re-verification cannot save it. `classifySubject` therefore buckets `byOwner` only when the compiled program's `regexp.LiteralPrefix()` contains a `/`, so every string the pattern can match provably starts with `owner/` and hashes to the bucket `candidatesFor` looks in. Inferring the owner from the raw pattern text before the first `/` (pre-2.1.0) was unsound: a quantified first slash (`owner/?repo`, `owner/*repo`) also matches slash-less subjects, so the mapping was dropped from the scan — see `TestIndexParity`.
- `AuthorizeRoles(issuer, subject, claims)` — issuer-bound authorization lookup via the index; `FindSessionPolicy(issuer, subject, role, claims)` — applies the **same** predicates as `AuthorizeRoles` (subject match + conditions satisfied) **plus** `slices.Contains(mapping.Roles, role)`, so a role's scoping policy always comes from the mapping that authorized it; among qualifying mappings the lowest `order` (first-declared) wins, and `(nil, nil)` means no mapping grants the role (e.g. tag-auth, which by design carries no session policy). Resolving by subject alone — the pre-2.1.0 behavior — let an earlier broad mapping strip a privileged role's policy. `IssuerSessionTags(issuer)` returns the per-issuer `session_tags` spec consumed by `aws.BuildSessionTags`.
- `TagAuth` (opt-in, default off) + `Authorize(roleTags, claims, verifiedIssuer, subject)` in `tagauth.go` — tag-based authorization fallback; exact matching only (AWS tag charset, no regex), space-list = OR, cross-tag = AND. Identity gate: the canonical `<prefix>subject` tag, or the legacy `<prefix>repo`/`<prefix>repo-owner` aliases (retained through the v2 migration window). Issuer gate: once more than one issuer is configured (`multiIssuer`), a role must also carry a matching `<prefix>issuer` tag or it fails closed (no cross-issuer identity collision); with a single issuer the tag is optional but still checked if present. Full field set: `TagPrefix string` (default `"aow/"`), `DefaultOrg string` (expands bare `aow/repo` tokens to `<org>/name`; must not contain `/` or whitespace), `TransitiveSessionTags bool` (marks every attached session tag — the issuer's `session_tags` spec — immutable through role chaining, default false). See `docs/TAG_BASED_AUTHORIZATION.md`.
- `CrossAccount` (opt-in, default off) — a **policy gate** for member-account role assumption: `Enabled bool` false (or the struct nil) hard-blocks every cross-account operation (both `AssumeRole` and, for `TagAuth`, `GetRoleTags`) — enforced in `internal/aws`, not here. Role assumption is always direct hub→target, one hop, using the warden's own credentials; `SpokeRoleName string` (default `"aow-spoke"`) + `ExternalID string` (optional) only configure the separate hub→spoke hop `GetRoleTags` uses to read a target role's IAM tags cross-account for `TagAuth` — never the assume path. `SpokeSessionDuration time.Duration` (default 15m) bounds that hop's session. `AllowedAccounts []string` (allow-list of member account IDs; hub implicit; empty = any once enabled; non-12-digit rejected by `Validate()`). Env: `AOW_CROSS_ACCOUNT_*`. See `docs/examples/cross-account/`.
- `Provider` — hot-reload wrapper around `Config`: `NewProvider(base, interval, format, fetch, ...ProviderOption)` / `NewStaticProvider(cfg)`; `MaybeRefresh(ctx)` does a lazy per-request reload (atomic swap from clone of pristine base) and re-resolves `config_fragments` on the same cadence; `Get()` returns the current active config. The token validator calls `Get()` on every `Validate()` so issuer/audience changes apply immediately.
- `config_fragments` (`fragments.go`) — additional sources merged onto the base config's `default_issuer`/`role_sets`/`role_mappings`/`role_groups` only (any other key is a hard error); local paths are content-hashed (sha256) for change detection and read directly; `WithFragmentFetcher(FragmentFetchFunc)` is the seam for remote (`"scheme://"`) sources, e.g. `s3://` — a failed/invalid fragment retains the last-good config.

## Gotchas

- Never compile regex per request — only in `Validate()`.
- **Never compile a `*Condition` that another snapshot may be reading.** `compileCondition` mutates in place (`cond.compiled = cond.compiled[:0]`, then re-appends), and one `*Condition` can be shared across snapshots — a `config_fragment` reuses its parsed struct across hot reloads, and a `role_group` shares one `Defaults.Conditions` across every expanded subject. `appendEffective` therefore calls `cloneCondition(m.Conditions)` **before** `compileCondition`, giving each effective mapping private memory. Removing that clone reintroduces an authorization bypass: `Get()` readers hit `satisfiesConditions` unlocked, and a transiently-empty `compiled` list makes **all conditions pass** (regression test: `TestHotReloadConditionRace`). `appendEffective` takes `RoleMapping` **by value** — keep it that way, or the clone writes back into the shared struct.
- Keep patterns specific; reject `.*` for security conditions.
- Env keys are case-insensitive; config keys are snake_case.
- `WithFragmentFetcher` is defined but **not called from any production bootstrap path** — `internal/handler/bootstrap.go`'s `buildConfigProvider()` passes no `ProviderOption`s, so a `"scheme://"` `config_fragments` entry hard-fails to fetch in the shipped binaries today. Only local filesystem-path fragments work until a fetcher is wired in.
